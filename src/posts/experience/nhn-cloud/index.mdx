### AWX 도입 리서치

#### 배경

내부에서는 Ansible을 Python 코드로 직접 호출해 인프라 작업을 자동화하고 있었다. Playbook 실행 자체는 안정적이었지만, 스케줄링·실행 이력·모니터링을 한곳에서 관리하기 어려웠다. AWX(Ansible Automation Platform의 오픈소스 UI) 도입이 이 문제를 해소할 수 있는지 검토하기로 했다.

#### 담당

- 기존 Python + Ansible 운영 방식과 AWX의 역할·기대 효과 조사
- AWX Operator 기반 Kubernetes 배포 모델 리서치
- 파일럿 클러스터에 AWX를 띄워 GUI 기반 Job 실행·스케줄링 검증

#### 핵심 구현

- AWX Controller → Ansible Playbook → 대상 노드 흐름을 K8s 파일럿 환경에서 재현
- Operator CRD로 AWX 인스턴스를 선언적으로 배포하고, 기존 Playbook을 AWX Job Template에 연결
- 스케줄 Job, 작업 로그, RBAC 등 운영 시나리오별 체크리스트 작성

### Vault HA 버전 검증

#### 배경

Vault를 Active-Standby HA로 운영하려면 노드 간 버전 정합성이 중요하다. 업그레이드·롤백 과정에서 의도치 않게 버전이 어긋나면 복제 지연이나 failover 장애로 이어질 수 있었고, 서로 다른 버전의 Vault가 HA 옵션으로 실제로 동작하는지에 대한 사전 검증이 필요했다.

#### 담당

- Vault Active-Standby HA 클러스터를 별도 PoC 환경에 구성
- 노드 간 Vault 버전을 의도적으로 다르게 맞춘 뒤 복제·failover·롤백 시나리오 재현
- 시나리오별 관측 항목과 복구 절차를 체크리스트로 정리

#### 핵심 구현

- 정상 동기화 / minor 버전 차 / major 버전 차 세 가지 시나리오를 분리해 테스트
- Active ↔ Standby 복제 상태, seal·unseal, leader 선출 과정을 단계별로 기록
- 버전별 동작 차이를 PoC 노트로 정리

### OPA 리서치

#### 배경

Zero Trust 기반 보안 상품을 검토하려면 정책을 선언적으로 정의하고 런타임에 평가하는 엔진이 필요했다. Gatekeeper 같은 응용 사례를 바로 도입하기보다, OPA(Open Policy Agent) 자체가 제공하는 기능을 이해하는 것이 먼저라고 판단했다.

#### 담당

- Rego로 샘플 정책 작성 및 `opa eval`로 평가 흐름 확인
- Bundle로 정책을 묶어 배포·로드하는 방식 실습
- REST API, Decision Log 등 OPA 핵심 기능을 하나씩 써보며 동작 정리

#### 핵심 구현

- `opa run`으로 로컬에서 정책 서버를 띄우고 입력 JSON에 대한 allow/deny 결과 확인
- `opa build`로 Rego 정책을 Bundle 아티팩트로 패키징하고, OPA에 로드해 정책 교체 흐름 검증
- 보안 상품에 쓸 수 있는지 판단하기 위해 기능별(평가 API, Bundle 배포, 로깅) 메모 정리
