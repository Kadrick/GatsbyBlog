## 개요

- **역할:** 인턴 — 클라우드 인프라 검증 및 아키텍처 PoC
- **기간:** 2025.07 ~ 2025.12

Ansible 운영 자동화(AWX), 시크릿 관리(Vault HA), Zero Trust 정책(Rego/OPA) 세 영역에서 **독립적인 PoC·리서치**를 수행했다. 각 과제는 서로 다른 목적과 환경에서 진행되었다.

<CaseStudy
  id="nhn-awx-research"
  title="Case 1. AWX 도입 검토를 위한 기술 리서치 및 파일럿 검증"
  problem={
    <>
      Ansible 운영이 터미널·수동 실행에 의존해 스케줄링·모니터링·작업 이력 추적이
      어려웠고, AWX 도입 시 기대 효과와 운영 부담에 대한 근거가 부족했다.
    </>
  }
  action={
    <>
      AWX와 Ansible Operator 기반 배포 모델을 조사하고, Kubernetes 파일럿
      환경을 구성해 GUI 기반 작업 실행·스케줄링을 검증했다. 백오피스
      관리 코드에는 go test 기반 단위 테스트를 도입해 자동화 신뢰성도 함께
      점검했다.
    </>
  }
  application={
    <>
      파일럿 클러스터에서 AWX Controller → Ansible Playbook → 대상 노드
      흐름을 재현하고, 운영 시나리오(스케줄 Job, 작업 로그, RBAC)별
      체크리스트를 작성했다.
    </>
  }
  result={
    <>
      AWX 도입 여부를 판단할 수 있는 기술 리포트와 파일럿 결과를 정리했고,
      인프라 운영 자동화 로드맵 논의에 활용 가능한 근거를 마련했다.
    </>
  }
/>

<CaseStudy
  id="nhn-vault-ha-version"
  title="Case 2. Vault HA 구성을 위한 버전 충돌 시나리오 검증"
  problem={
    <>
      Vault를 Active-Standby HA로 운영할 때, 노드 간 버전 불일치가 발생하면
      복제·failover 과정에서 장애로 이어질 수 있었고, 업그레이드·롤백
      절차에 대한 사전 검증 체계가 없었다.
    </>
  }
  action={
    <>
      Vault Active-Standby HA 클러스터를 별도 PoC 환경에 구성하고, 의도적으로
      버전을 다르게 맞춘 뒤 복제 지연·failover·롤백 시나리오를
      재현·기록했다.
    </>
  }
  application={
    <>
      시나리오별(정상 동기화 / minor 버전 차 / major 버전 차) 관측 항목과
      복구 절차를 체크리스트로 정리해, 향후 프로덕션 HA 설계 시 참고
      자료로 활용했다.
    </>
  }
  result={
    <>
      버전 정합성을 HA 운영의 전제 조건으로 명시할 수 있는 근거를 확보했고,
      무중단 업그레이드 전략 수립에 필요한 실측 데이터를 제공했다.
    </>
  }
/>

<Mermaid chart={`flowchart TB
  subgraph vault [Vault HA PoC — 별도 환경]
    Active[Vault Active vN]
    Standby[Vault Standby vN-1]
    Test[버전 충돌 시나리오 테스트]
    Failover[Failover / 롤백 검증]
  end

  Active <-->|복제| Standby
  Test --> Active
  Test --> Standby
  Failover --> Active`} />

<CaseStudy
  id="nhn-zero-trust-rego"
  title="Case 3. Zero Trust 상품 방향성 검증 — Rego 기반 OPA 클라이언트 PoC"
  problem={
    <>
      Zero Trust 상품에 필요한 동적 정책 정의·평가·클라이언트 연동 구조가
      없었고, Rego 정책을 실제 워크로드에 적용했을 때의 동작과 운영
      복잡도를 사전에 알 수 없었다.
    </>
  }
  action={
    <>
      OPA를 정책 엔진으로 두고 Rego로 접근 제어·네트워크 정책 샘플을
      작성했다. 별도 K8s PoC 클러스터에 OPA/Gatekeeper 클라이언트를
      배포해 정책 평가·위반 차단 흐름을 검증했다.
    </>
  }
  application={
    <>
      정책 요청 → OPA Rego 평가 → allow/deny 결과 → 클라이언트(Admission /
      Sidecar) 반영 파이프라인을 구성하고, 대표 유스케이스 3종을 PoC
      시나리오로 문서화했다.
    </>
  }
  result={
    <>
      Zero Trust 상품 기획·개발에 참고할 수 있는 Rego/OPA 레퍼런스
      아키텍처와 정책 작성 가이드를 확보했다.
    </>
  }
/>

  end
