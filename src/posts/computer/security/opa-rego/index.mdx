[NHN Cloud 인턴](/post/experience/nhn-cloud) 기간에 보안 상품 방향을 검토하면서 OPA(Open Policy Agent)를 리서치했다. Gatekeeper 같은 응용 사례를 바로 붙이기보다, Bundle·평가 API 등 OPA가 제공하는 기능을 하나씩 써보며 정리했다.

---

## OPA란

OPA는 정책을 코드(Policy as Code)로 작성하고, JSON 형태의 입력에 대해 allow/deny를 판단하는 범용 엔진이다. Kubernetes, API Gateway, CI/CD 등 어디에든 붙일 수 있지만, 핵심은 항상 같다. Rego로 정책을 쓰고, OPA가 평가한다.

| 특징 | 설명 |
| --- | --- |
| 범용성 | 입력이 JSON이면 K8s manifest, HTTP 요청, Terraform plan 모두 평가 가능 |
| 분리 | 애플리케이션 로직과 정책 로직을 분리 |
| 선언적 | Rego로 "무엇을 허용할지"를 선언 |
| 결정론적 | 같은 입력 → 같은 결과 |

<Mermaid chart={`flowchart LR
  Input["입력 JSON"] --> OPA["OPA Engine"]
  Policy["Rego Policy"] --> OPA
  OPA --> Result["allow / deny / data"]`} />

OPA는 정책을 강제하지 않는다. 앞단 서비스나 클라이언트가 OPA의 결정을 받아 차단·허용을 실행한다.

---

## Rego란

Rego는 OPA 전용 선언적 정책 언어다. `input`으로 들어온 JSON을 기준으로 규칙을 평가한다.

```rego
package example

default allow = false

allow {
  input.method == "GET"
  input.path = "/public"
}
```

| 문법 요소 | 설명 |
| --- | --- |
| `package` | 정책 네임스페이스 |
| `default` | 규칙이 매칭되지 않을 때의 기본값 |
| `allow { ... }` | 조건이 모두 참이면 allow를 true로 설정 |
| `input` | 평가 대상 JSON |

CLI로 바로 확인할 수 있다.

```bash
opa eval -d policy.rego -i input.json "data.example.allow"
```

리서치 초반에는 `opa eval`로 Rego 문법과 평가 결과를 맞춰보는 데 시간을 썼다.

---

## OPA 서버와 REST API

`opa run`으로 정책 서버를 띄우면 HTTP API로 평가를 요청할 수 있다.

```bash
opa run --server policy.rego
```

```bash
curl -X POST localhost:8181/v1/data/example/allow \
  -d '{"input": {"method": "GET", "path": "/public"}}'
```

| API | 용도 |
| --- | --- |
| `POST /v1/data/{path}` | 특정 정책 규칙 평가 |
| `POST /v1/compile` | 정책을 부분 평가해 필터 조건 생성 (고급) |
| `GET /v1/policies` | 로드된 정책 조회 |

애플리케이션이나 사이드카가 OPA를 호출하는 구조는 이 REST API를 기반으로 한다.

## Bundle — 정책 패키징과 배포

Bundle은 Rego 정책과 메타데이터를 하나의 아티팩트로 묶어 OPA에 배포하는 방식이다. 리서치에서 가장 많이 다뤘던 기능이다.

```bash
opa build -b ./policies/ -o bundle.tar.gz
```

| 구성 요소 | 설명 |
| --- | --- |
| `.rego` 파일 | 정책 소스 |
| `data.json` | 정책에서 참조할 정적 데이터 |
| `.manifest` | Bundle 메타데이터 (선택) |

Bundle을 OPA에 로드하는 방법은 여러 가지다.

| 방식 | 설명 |
| --- | --- |
| `--bundle` 플래그 | `opa run --server --bundle bundle.tar.gz` |
| 파일 시스템 | OPA가 주기적으로 Bundle 디렉터리를 폴링 |
| HTTP 서버 | OPA가 원격 URL에서 Bundle을 pull |

<Mermaid chart={`flowchart LR
  Rego["Rego 소스"] --> Build["opa build"]
  Build --> Bundle["bundle.tar.gz"]
  Bundle -->|"로드 / pull"| OPA["OPA Server"]
  OPA --> Eval["정책 평가"]`} />

정책을 애플리케이션과 분리해 배포할 수 있다는 점이 보안 상품 관점에서 중요했다. 코드 배포 없이 Bundle만 교체하면 정책을 갱신할 수 있다.

## 그밖에 살펴본 기능

| 기능 | 설명 |
| --- | --- |
| Decision Log | 평가 요청·결과를 로그로 남겨 감사·디버깅에 활용 |
| `opa test` | Rego 단위 테스트 작성·실행 |
| Data API | `data` 경로에 정적·동적 데이터를 두고 정책에서 참조 |

```bash
opa test ./policies/
```

`opa test`로 샘플 입력에 대한 allow/deny를 자동 검증하는 흐름도 확인했다.

---

## 어디에 쓰이는가

리서치에서는 응용 사례를 직접 구축하지 않았지만, OPA가 쓰이는 대표 지점은 다음과 같다.

| 영역 | 예시 |
| --- | --- |
| Kubernetes | Gatekeeper 등 Admission Controller에서 리소스 생성 시 정책 평가 |
| API 인가 | Envoy, API Gateway에서 HTTP 요청 allow/deny |
| CI/CD | Terraform plan, Dockerfile, manifest 사전 검사 |

공통적으로 "클라이언트가 입력 JSON을 OPA에 넘기고, 결정을 받아 처리한다"는 패턴이 반복된다. Bundle과 REST API를 이해하면 이 응용 사례들도 같은 모델 위에 있다.

---

## 정리

OPA는 Rego 정책을 평가하는 엔진이고, `opa eval`·REST API로 결과를 확인한다. Bundle은 정책을 패키징해 배포하는 핵심 메커니즘이며, 정책을 애플리케이션과 분리해 운영할 수 있게 해준다. 보안 상품을 검토할 때는 Gatekeeper 같은 통합 제품보다, 이 기본 기능들을 먼저 이해하는 편이 도움이 됐다.
