[KNU Online Judge](/post/projects/judge)를 운영하면서 기존 시스템의 채점 데이터를 통합하는 과정에서 제출·채점 테이블의 row 수가 급격히 늘었다. 단일 테이블로는 조회·유지보수 부담이 커져 **월별 RANGE 파티셔닝**을 검토하고 도입했다. 이 글에서는 PostgreSQL 파티셔닝의 개념과 방식, 월별 파티션 설계 시 알아둘 점을 정리한다.

---

## 파티셔닝이란

파티셔닝(Partitioning)은 **논리적으로 하나인 큰 테이블을 물리적으로 여러 조각으로 나누는 것**이다. 애플리케이션 입장에서는 여전히 하나의 테이블처럼 조회·삽입하지만, 실제 데이터는 파티션(partition)이라는 자식 테이블에 분산 저장된다.

PostgreSQL 공식 문서가 제시하는 주요 이점은 다음과 같다.

| 이점 | 설명 |
| --- | --- |
| **조회 성능** | 자주 조회하는 최근 데이터가 소수 파티션에 모이면, 인덱스·캐시 효율이 좋아진다 |
| **스캔 범위 축소** | 특정 파티션만 sequential scan하면 전체 테이블을 훑지 않아도 된다 |
| **유지보수 용이** | 오래된 파티션을 `DROP TABLE` 또는 `DETACH PARTITION`으로 빠르게 제거 |
| **스토리지 분리** | 파티션별로 서로 다른 tablespace를 지정할 수 있다 |

일반적으로 테이블 크기가 DB 서버 물리 메모리를 넘어설 때 파티셔닝 효과가 두드러진다. Online Judge처럼 학기마다 제출이 누적되는 시계열 데이터에는 특히 잘 맞는다.

## PostgreSQL 선언적 파티셔닝

PostgreSQL 10부터 **선언적 파티셔닝(Declarative Partitioning)** 을 기본 지원한다. `CREATE TABLE` 시 `PARTITION BY` 절로 분할 방식과 파티션 키를 선언하고, 자식 파티션을 `PARTITION OF`로 연결한다.

부모 테이블은 데이터를 직접 저장하지 않는 **가상 테이블**이고, 실제 row는 각 파티션에 저장된다. `INSERT` 시 파티션 키 값에 따라 자동으로 적절한 파티션으로 라우팅된다.

### 지원하는 파티셔닝 방식

PostgreSQL은 세 가지 방식을 제공한다. ([공식 문서 — Table Partitioning](https://www.postgresql.org/docs/current/ddl-partitioning.html))

| 방식 | 설명 | 대표 사용 사례 |
| --- | --- | --- |
| **RANGE** | 키 값의 범위로 분할. 하한은 포함, 상한은 미포함 | 날짜·시간 기반 로그, 제출 이력 |
| **LIST** | 키 값 목록을 명시해 분할 | 지역 코드, 상태값(`active` / `archived`) |
| **HASH** | 해시 나머지(modulus)로 균등 분할 | 파티션 키에 자연스러운 범위가 없을 때 |

Online Judge의 채점 데이터는 **제출 시각(`created_at`) 기준으로 월 단위 조회**가 많고, 오래된 학기 데이터를 통째로 내리는 요구도 있어 **RANGE 파티셔닝**이 가장 자연스러운 선택이었다.

## RANGE 파티셔닝 — 월별 파티션 예시

공식 문서의 `measurement` 예시와 같은 패턴으로, 제출 테이블을 월별로 나누는 기본 구조는 다음과 같다.

```sql
-- 부모(파티션) 테이블
CREATE TABLE submissions (
    id          BIGSERIAL,
    user_id     BIGINT      NOT NULL,
    problem_id  BIGINT      NOT NULL,
    status      TEXT        NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),

    -- PK는 반드시 파티션 키를 포함해야 함
    PRIMARY KEY (created_at, id)
) PARTITION BY RANGE (created_at);
```

```sql
-- 월별 자식 파티션
CREATE TABLE submissions_2024_08 PARTITION OF submissions
    FOR VALUES FROM ('2024-08-01') TO ('2024-09-01');

CREATE TABLE submissions_2024_09 PARTITION OF submissions
    FOR VALUES FROM ('2024-09-01') TO ('2024-10-01');

CREATE TABLE submissions_2024_10 PARTITION OF submissions
    FOR VALUES FROM ('2024-10-01') TO ('2024-11-01');
```

RANGE 경계는 **하한 이상, 상한 미만**이다. `TO ('2024-09-01')`이면 9월 1일 00:00:00은 다음 파티션에 속한다.

  P1["submissions_2024_08\n(pruned)"]
  P2["submissions_2024_09\n(scanned)"]
  P3["submissions_2024_10\n(pruned)"]

  Q --> P2`} />

### 파티션 키와 PRIMARY KEY

파티션 테이블에 `UNIQUE` 또는 `PRIMARY KEY`를 걸려면 **제약 조건 컬럼에 파티션 키가 반드시 포함**되어야 한다. 파티션마다 독립된 인덱스만 존재하기 때문에, 파티션 키 없이 `id`만으로 유일성을 보장할 수 없기 때문이다.

그래서 시계열 테이블에서는 `(created_at, id)` 형태의 복합 PK가 흔하다.

### 인덱스

부모 테이블에 인덱스를 만들면 **기존·향후 파티션에 자동으로 동일 인덱스가 생성**된다.

```sql
CREATE INDEX ON submissions (user_id, created_at);
CREATE INDEX ON submissions (problem_id, created_at);
```

## 파티션 프루닝 (Partition Pruning)

파티션 프루닝은 쿼리의 `WHERE` 조건을 보고 **관련 없는 파티션을 실행 계획에서 제외**하는 최적화다. `enable_partition_pruning` 설정이 켜져 있어야 하며(PostgreSQL 기본값 `on`), 쿼리가 파티션 키로 범위를 한정할 때 효과가 난다.

```sql
-- 2024년 9월 제출만 조회 → submissions_2024_09 파티션만 스캔
SELECT *
FROM submissions
WHERE created_at >= '2024-09-01'
  AND created_at <  '2024-10-01'
  AND user_id = 42;
```

`EXPLAIN`으로 실행 계획을 확인하면 다른 월 파티션이 계획에서 빠진 것을 볼 수 있다.

주의할 점:

- `WHERE` 절에서 파티션 키에 **함수나 타입 캐스트**를 걸면 프루닝이 비활성화될 수 있다. (`created_at::date = '2024-09-15'` 대신 범위 조건 사용)
- 프루닝은 파티션 키의 **제약 조건**에 기반한다. 인덱스 유무와는 별개다.

## LIST · HASH 파티셔닝

### LIST

특정 값 집합으로 나눈다. 지역·상태처럼 이산적인 키에 적합하다.

```sql
CREATE TABLE orders (
    id          BIGINT,
    region      TEXT NOT NULL,
    amount      NUMERIC,
    PRIMARY KEY (region, id)
) PARTITION BY LIST (region);

CREATE TABLE orders_kr PARTITION OF orders
    FOR VALUES IN ('KR');
CREATE TABLE orders_us PARTITION OF orders
    FOR VALUES IN ('US');
```

### HASH

`HASH` 파티션 키를 modulus로 나눈 나머지로 분배한다. 키 값에 자연스러운 시간·범위 구간이 없고, 균등 분산이 목적일 때 쓴다.

```sql
CREATE TABLE events (
    id          BIGINT,
    payload     JSONB,
    PRIMARY KEY (id)
) PARTITION BY HASH (id);

CREATE TABLE events_p0 PARTITION OF events
    FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE events_p1 PARTITION OF events
    FOR VALUES WITH (MODULUS 4, REMAINDER 1);
-- ... p2, p3
```

Online Judge처럼 **최근 학기 데이터 위주로 조회하고, 오래된 학기 데이터를 통째로 내리는** 패턴에는 HASH보다 RANGE가 맞다.

## 기존 테이블에서 파티션으로 전환

PostgreSQL은 **일반 테이블을 파티션 테이블로 직접 변환하는 명령은 없다.** 대신 아래 순서로 마이그레이션한다.

1. 파티션 부모 테이블을 새로 생성
2. 기간별로 자식 파티션 생성
3. 기존 데이터를 기간별로 나눠 각 파티션에 적재
4. 또는 기존 테이블을 `ATTACH PARTITION`으로 연결

`ATTACH PARTITION` 방식은 데이터를 먼저 별도 테이블에 넣고 검증한 뒤 부모에 붙이므로, 운영 중인 서비스에 유리하다.

```sql
-- 1. 기간별 데이터를 담을 독립 테이블 생성
CREATE TABLE submissions_2024_08
  (LIKE submissions INCLUDING DEFAULTS INCLUDING CONSTRAINTS);

-- 2. 파티션 범위 CHECK 제약 추가 (ATTACH 시 전체 스캔 방지)
ALTER TABLE submissions_2024_08 ADD CONSTRAINT y2024m08
  CHECK (created_at >= TIMESTAMPTZ '2024-08-01'
     AND created_at <  TIMESTAMPTZ '2024-09-01');

-- 3. 레거시 데이터 적재
INSERT INTO submissions_2024_08
SELECT * FROM legacy_submissions
WHERE created_at >= '2024-08-01'
  AND created_at <  '2024-09-01';

-- 4. 파티션으로 연결
ALTER TABLE submissions ATTACH PARTITION submissions_2024_08
    FOR VALUES FROM ('2024-08-01') TO ('2024-09-01');
```

Online Judge에서는 **기존 시스템 채점 데이터를 월별로 나눠 적재한 뒤 ATTACH**하는 방식으로 통합했다.

## 파티션 운영 — 추가 · 제거

파티셔닝의 핵심 장점 중 하나는 **데이터 이동 없이 파티션 구조만 조작**해 대량 데이터를 다룰 수 있다는 점이다.

### 새 월 파티션 추가

```sql
CREATE TABLE submissions_2026_04 PARTITION OF submissions
    FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');
```

매월 파티션을 수동으로 만들기 번거롭다면 [pg_partman](https://github.com/pgpartman/pg_partman) 확장으로 자동화할 수 있다.

```sql
SELECT partman.create_parent(
    p_parent_table => 'public.submissions',
    p_control      => 'created_at',
    p_type         => 'native',
    p_interval     => '1 month',
    p_premake      => 3   -- 앞으로 3개월치 파티션 미리 생성
);
```

### 오래된 파티션 제거

```sql
-- 즉시 삭제
DROP TABLE submissions_2023_01;

-- 분리 후 백업·아카이브 (PG 14+ CONCURRENTLY로 락 최소화)
ALTER TABLE submissions DETACH PARTITION submissions_2023_01 CONCURRENTLY;
```

`DELETE`로 수백만 row를 지우는 것보다 훨씬 빠르고, `VACUUM` 부담도 적다.

## 설계 시 주의할 점

### 파티션 수

파티션이 너무 적으면 파티션 하나가 여전히 크고, 너무 많으면 쿼리 플래닝 비용과 메모리 사용이 늘어난다. 공식 문서 기준으로 **수천 개 이하**면 일반적으로 문제없고, 월별 파티션이면 수년치 데이터도 수십 개 수준으로 관리 가능하다.

### DEFAULT 파티션

범위에 맞지 않는 row를 받을 `DEFAULT` 파티션을 둘 수 있지만, 프루닝 효율이 떨어질 수 있어 신중히 사용한다. 월별 운영에서는 **미리 다음 달 파티션을 만들어 두는 것**이 낫다.

### 조회 패턴과 파티션 키

파티션 키는 **실제 쿼리가 자주 필터하는 컬럼**이어야 한다. Online Judge에서는 강의·학기별 조회가 `created_at` 범위와 맞아떨어져 월별 RANGE가 효과적이었다.

## Online Judge에서의 적용

기존 시스템 채점 데이터를 신규 Online Judge DB로 통합하면서 `submissions` 테이블 row 수가 급증했다. 단일 테이블 상태에서는 인덱스 크기와 sequential scan 범위가 커져 조회가 느려졌고, 학기가 지난 데이터를 정리하는 비용도 컸다.

월별 RANGE 파티셔닝을 도입한 뒤:

- 최근 학기 조회는 해당 월 파티션만 스캔
- 레거시 데이터는 기간별로 나눠 `ATTACH PARTITION`으로 통합
- 오래된 학기 데이터는 `DETACH` 또는 `DROP`으로 정리

실제 적용 맥락과 작업 내역은 [KNU Online Judge](/post/projects/judge) 프로젝트 글에서 확인할 수 있다.
