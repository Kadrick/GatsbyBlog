[NHN Cloud 인턴](/post/experience/nhn-cloud) 기간에 Vault HA 구성을 검토하면서, Active-Standby 클러스터에서 노드 간 버전이 다를 때 어떤 일이 벌어지는지 PoC로 확인했다. 이 글에서는 Vault HA의 기본 구조와 버전 정합성이 왜 중요한지 정리한다.

---

## Vault란

HashiCorp Vault는 **시크릿(비밀번호, API 키, 인증서 등)을 중앙에서 저장·발급·회전**하는 도구다. 애플리케이션은 Vault API를 통해 필요한 시점에만 시크릿을 받아 쓴다.

| 개념 | 설명 |
| --- | --- |
| Secret Engine | KV, Database, PKI 등 시크릿 종류별 백엔드 |
| Auth Method | 토큰, AppRole, Kubernetes 등 인증 방식 |
| Policy | 경로 단위 접근 권한 정의 |
| Seal / Unseal | 마스터 키로 Vault 데이터를 암호화·해제하는 과정 |

Vault는 기본적으로 **sealed** 상태로 시작한다. Unseal 키(또는 auto-unseal 설정)로 풀어야 API가 동작한다.

## HA 모드

단일 Vault 인스턴스는 장애 시 시크릿 발급이 중단된다. 프로덕션에서는 **HA(High Availability)** 구성을 쓴다.

### Active-Standby

가장 흔한 형태다. 한 노드가 **Active(leader)** 로 쓰기를 처리하고, 나머지 **Standby** 노드는 복제를 따라간다.

<Mermaid chart={`flowchart LR
  Client["Client"] --> Active["Vault Active"]
  Active <-->|"복제"| Standby1["Vault Standby"]
  Active <-->|"복제"| Standby2["Vault Standby"]`} />

- 읽기: Standby에서도 가능한 경우가 많음 (설정·버전에 따라 다름)
- 쓰기: Active 노드만 처리
- Failover: Active 장애 시 Standby 중 하나가 leader로 승격

### Storage Backend

HA 동작은 스토리지 백엔드에 따라 달라진다.

| 백엔드 | 특징 |
| --- | --- |
| Integrated Storage (Raft) | Vault 1.4+ 내장, 별도 Consul 없이 HA 구성 가능 |
| Consul | 전통적인 HA 구성, Consul 클러스터 별도 운영 |
| 외부 DB (PostgreSQL 등) | 엔터프라이즈 환경에서 사용, Vault Enterprise 기능 |

PoC에서는 Integrated Storage(Raft) 기반 Active-Standby를 구성했다.

---

## Seal, Leader 선출, Failover

HA 클러스터에서 장애가 나면 다음 순서로 처리된다.

1. Active 노드 응답 불가 감지
2. Standby 노드들이 leader 선출(Raft quorum 기반)
3. 새 Active가 unsealed 상태면 서비스 재개
4. 클라이언트는 동일 엔드포인트(또는 로드밸런서)로 재연결

| 단계 | 확인 포인트 |
| --- | --- |
| 복제 지연 | Standby가 Active의 Raft 로그를 따라가는지 |
| Unseal 상태 | failover 후 새 Active가 sealed 상태가 아닌지 |
| API 응답 | `vault status`, `vault operator raft list-peers` |

## 버전 불일치 시나리오

업그레이드·롤백 과정에서 클러스터 노드마다 Vault 버전이 잠깐 어긋날 수 있다. PoC에서는 이 상황을 의도적으로 재현했다.

| 시나리오 | 관측 내용 |
| --- | --- |
| 동일 버전 | 정상 복제, failover 성공 |
| Minor 버전 차 (예: 1.15.x ↔ 1.15.y) | 대체로 동작하나, HashiCorp 권장은 클러스터 전체 동일 버전 |
| Major 버전 차 (예: 1.14 ↔ 1.15) | 복제 오류, leader 선출 실패, API 호환성 문제 가능 |

HashiCorp는 HA 클러스터 내 **모든 노드를 동일 버전으로 유지**할 것을 권장한다. 버전이 다르면 Raft 프로토콜·스토리지 포맷·API 동작이 맞지 않아 복제가 끊기거나 failover가 실패할 수 있다.

### 안전한 업그레이드 순서 (요약)

1. Standby 노드를 하나씩 최신 버전으로 교체
2. 각 교체 후 복제·상태 정상 확인
3. 마지막에 Active를 failover로 Standby로 내린 뒤 교체
4. 전체 노드 버전 일치 확인

롤백도 같은 원칙이다. 클러스터에 서로 다른 major 버전이 공존하는 상태는 피한다.

---

## 정리

Vault HA는 Active-Standby와 Raft(또는 Consul) 복제로 가용성을 확보한다. 운영에서 흔히 놓치는 부분은 버전 정합성인데, 노드 간 버전이 어긋나면 복제 장애나 failover 실패로 이어질 수 있다. 업그레이드·롤백 시에는 Standby부터 순차 교체하고, 전 노드가 같은 버전인지 확인하는 절차가 필요하다.
