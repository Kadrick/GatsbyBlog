## Ansible이란

Ansible은 SSH(또는 API)를 통해 원격 호스트에 명령을 실행하는 **구성 관리·자동화 도구**다. 에이전트를 설치하지 않고, Playbook(YAML)에 작업 순서를 선언하면 된다.

| 개념 | 설명 |
| --- | --- |
| Playbook | 실행할 작업 목록을 YAML로 작성한 파일 |
| Task | Playbook 안의 개별 작업 단위 |
| Module | `copy`, `yum`, `template` 등 Ansible이 제공하는 실행 단위 |
| Inventory | 대상 호스트 목록과 그룹 정의 |
| Idempotency | 같은 Playbook을 여러 번 실행해도 결과가 수렴하는 성질 |

```yaml
# 예시: nginx 설치 및 서비스 기동
- hosts: web
  tasks:
    - name: Install nginx
      ansible.builtin.yum:
        name: nginx
        state: present
    - name: Start nginx
      ansible.builtin.service:
        name: nginx
        state: started
```

Playbook은 "무엇을 할지"만 선언하고, 실제 실행·상태 비교는 Ansible 엔진이 처리한다.

## Python에서 Ansible을 직접 호출하는 방식

Ansible은 CLI(`ansible-playbook`)뿐 아니라 Python API로도 실행할 수 있다. 백오피스나 내부 도구에서 Playbook을 트리거할 때 이 방식을 쓰는 경우가 많다.

| 방식 | 특징 |
| --- | --- |
| `subprocess`로 CLI 호출 | 구현이 단순하지만 출력 파싱·에러 처리를 직접 해야 함 |
| `ansible-runner` | Playbook 실행을 Python에서 감싸고, 이벤트 콜백·아티팩트 수집 지원 |
| `ansible-playbook` 모듈 임포트 | 세밀한 제어가 가능하지만 Ansible 내부 API에 의존 |

```python
import ansible_runner

result = ansible_runner.run(
    playbook="deploy.yml",
    inventory="inventory/hosts",
    extravars={"env": "staging"},
)

if result.status == "successful":
    print("완료")
else:
    print(result.stdout.read())
```

이 방식의 장점은 기존 애플리케이션 흐름 안에 자동화를 자연스럽게 끼워 넣을 수 있다는 점이다. 반면 스케줄링, 실행 이력 조회, 권한 분리, 웹 UI는 직접 만들거나 다른 도구에 맡겨야 한다.

## AWX란

AWX는 Ansible Automation Platform의 **오픈소스 UI·API 레이어**다. Ansible 엔진 위에 다음 기능을 올린다.

| 기능 | 설명 |
| --- | --- |
| Job Template | Playbook + Inventory + Credential을 묶어 재사용 가능한 실행 단위로 등록 |
| 스케줄링 | Cron 기반으로 Job을 주기 실행 |
| RBAC | 팀·역할별로 Template·Credential 접근 권한 분리 |
| 실행 이력 | 누가, 언제, 어떤 Playbook을 실행했는지 로그로 남김 |
| Inventory 관리 | 호스트 목록을 UI에서 관리하거나 외부 소스와 동기화 |

Python 직접 호출이 "애플리케이션에서 필요할 때 실행"에 가깝다면, AWX는 "조직 단위로 자동화를 운영"하는 도구에 가깝다.

## AWX Operator로 Kubernetes에 배포하기

AWX는 단일 바이너리로도 띄울 수 있지만, Kubernetes 환경에서는 **AWX Operator**로 선언적으로 관리하는 편이 일반적이다. Operator가 Custom Resource를 보고 AWX 인스턴스·PostgreSQL·서비스를 프로비저닝한다.

```yaml
apiVersion: awx.ansible.com/v1beta1
kind: AWX
metadata:
  name: awx-pilot
spec:
  service_type: ClusterIP
  ingress_type: ingress
```

파일럿에서는 이 흐름을 검증했다.

1. Operator를 클러스터에 설치
2. `AWX` CR로 Controller 인스턴스 생성
3. 기존 Playbook을 Project로 등록
4. Job Template을 만들어 GUI에서 실행·스케줄 테스트

## Python 직접 호출 vs AWX

| 항목 | Python 직접 호출 | AWX |
| --- | --- | --- |
| 통합성 | 기존 백오피스·CI와 바로 연동 | REST API·웹 UI 제공 |
| 운영 가시성 | 직접 로깅·모니터링 구현 필요 | 실행 이력·상태 UI 내장 |
| 스케줄링 | cron·앱 로직으로 처리 | 내장 스케줄러 |
| 운영 부담 | 가벼움 | Controller·DB 등 추가 컴포넌트 |
| 적합한 경우 | 앱 내부 트리거, 소규모 자동화 | 팀 단위 운영, 감사·권한 관리 필요 |

AWX가 항상 정답은 아니다. 이미 Python으로 안정적으로 돌아가는 자동화가 있다면, AWX는 모니터링·이력·권한 관리가 필요할 때 도입을 검토하면 된다.

---

## 정리

Ansible은 Playbook으로 인프라 작업을 선언적으로 자동화하는 도구이고, Python API로 애플리케이션에 직접 붙일 수 있다. AWX는 그 위에 UI·스케줄링·RBAC·실행 이력을 얹어 조직 단위 운영을 돕는다. K8s에서는 AWX Operator로 파일럿 환경을 빠르게 구성해 도입 여부를 판단할 수 있다.
