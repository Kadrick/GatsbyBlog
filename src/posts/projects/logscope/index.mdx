장애 대응 시 운영자는 에러 로그와 CPU·메모리·응답 지연 같은 메트릭을 동시에 봐야 원인을 좁힌다. 그런데 기존 접근은 로그 텍스트 분석에 치우치는 경우가 많아, 같은 에러 메시지라도 자원 고갈인지 네트워크 지연인지 구분하기 어렵다. **AI를 활용한 자동 로그 분석 및 대응**은 에러 로그(텍스트)와 시스템 메트릭(수치)을 함께 입력받아 장애 유형을 분류하는 멀티모달 신경망을 제안하고, 1인으로 해커톤 구현 부문 PoC까지 완성했다.

## 배경 · 문제 정의

운영 환경에서 장애가 발생하면 보통 아래 두 종류의 신호가 동시에 남는다.

- **에러 로그** — `SocketTimeoutException`, `OutOfMemoryError`처럼 증상을 설명하는 텍스트
- **시스템 메트릭** — 장애 시점의 CPU·메모리 사용률, 응답 지연(Latency)

문제는 로그 메시지만으로는 원인이 겹치는 경우가 많다는 점이다. 예를 들어 `Task timed out after 10.00 seconds`는 네트워크 지연일 수도 있고, CPU·메모리가 포화된 상태에서 처리가 밀려 발생한 **자원 고갈**일 수도 있다. 실제 온콜에서는 로그와 메트릭을 함께 보며 판단하지만, 자동화 도구는 로그 키워드 매칭에 그치는 경우가 많다.

이 프로젝트의 핵심 아이디어는 **"운영자가 실제로 하는 판단을 모델 구조에 반영한다"** 는 것이다. 텍스트 가지와 메트릭 가지를 분리해 학습한 뒤 병합함으로써, 동일한 로그라도 메트릭 문맥에 따라 다른 장애 유형을 추론할 수 있도록 설계했다.

## 작업 내용

### 과제 제안 · 데이터 설계

- **장애 유형 정의** — 자원 고갈, 디스크 부족, 연결 거부, 네트워크 타임아웃, DB 커넥션 풀 고갈, 인증 실패, 코드 로직 오류, 컨테이너 크래시 등 10개 클래스로 분류 체계 수립
- **학습 데이터 구성** — `ErrorLog`, `CPU_Usage(%)`, `Memory_Usage(%)`, `Latency(ms)`, `ErrorType` 컬럼을 가진 `error_logs.csv` (181건) 작성
- **시나리오 기반 샘플링** — OOM, 디스크 Full, DB 풀 고갈, SSL 인증서 오류, CrashLoopBackOff 등 실제 운영에서 자주 보는 패턴을 의도적으로 포함

### 데이터 전처리

- **결측치 처리** — 메트릭 컬럼의 `N/A` 값을 0으로 치환
- **텍스트 전처리** — `Tokenizer`로 로그를 시퀀스화하고, `max_len=60`, `vocab_size=2000`으로 패딩
- **메트릭 정규화** — `StandardScaler`로 CPU·메모리·Latency 스케일 통일
- **라벨 인코딩** — `LabelEncoder`로 10개 장애 유형을 정수 클래스로 변환
- **학습/검증 분리** — `train_test_split(test_size=0.2)` + Early Stopping

### 멀티모달 모델 구현

- **텍스트 가지 (Branch 1)** — `Embedding(32)` → `GlobalAveragePooling1D` → `Dense(24, relu)` → `Dropout(0.2)`
- **메트릭 가지 (Branch 2)** — `Dense(16, relu)` → `Dense(8, relu)`
- **병합 · 분류** — 두 가지 특징을 `concatenate`한 뒤 `Dense(24, relu)` → `softmax` 출력
- **학습 설정** — Adam 옵티마이저, `sparse_categorical_crossentropy`, batch size 16, Early Stopping(`patience=5`)

### 평가 · 시나리오 검증

- **홀드아웃 테스트** — 소규모 데이터셋(181건) 기준 테스트 정확도 약 48.7%
- **실전 시나리오 테스트** — 학습에 없는 조합도 포함해 6개 카테고리·20여 건의 장애 시나리오로 추론 품질 검증
- **멀티모달 구분력 확인** — 동일 로그 + 다른 메트릭 조합에서 서로 다른 장애 유형을 예측하는지 집중 검증

## 설계 포인트

### 왜 멀티모달인가

로그만 쓰면 **증상(symptom)** 은 잡히지만 **원인(root cause)** 은 놓치기 쉽다. 해커톤 PoC에서 가장 강조한 검증 케이스가 바로 이 부분이다.

| 시나리오 | 에러 로그 | CPU | Memory | Latency | 예측 결과 |
| --- | --- | --- | --- | --- | --- |
| Case A | `Task timed out after 10.00 seconds` | 95% | 98% | 10,000ms | **Resource_Exhaustion** (48.4%) |
| Case B | `Task timed out after 10.00 seconds` | 10% | 20% | 30,000ms | **Network_Timeout** (99.6%) |

로그 문구는 동일하지만, 자원 사용률이 높으면 자원 고갈 쪽으로, 자원은 여유 있는데 지연만 크면 네트워크 타임아웃 쪽으로 기울었다. 운영자가 메트릭을 보며 내리는 판단을 모델이 모방하는 형태다.

### 분류 대상 장애 유형

모델이 구분하도록 설계한 10개 클래스는 아래와 같다.

| 클래스 | 대표 시나리오 |
| --- | --- |
| `Resource_Exhaustion` | OOM, 메모리 할당 실패 |
| `Disk_Full` | `No space left on device` |
| `Connection_Refused` | DB·API 연결 거부 |
| `Network_Timeout` | 소켓 읽기 타임아웃, 외부 API 무응답 |
| `DB_Pool_Exhaustion` | HikariCP 커넥션 풀 고갈 |
| `Auth_Failure` | 401/403, API Key·권한 오류 |
| `Code_Logic_Error` | NullPointerException 등 애플리케이션 버그 |
| `Container_Crash` | CrashLoopBackOff, 모듈 누락 |
| `Data_Format_Error` | 데이터 파싱·포맷 오류 |
| `ErrorType` | (데이터 라벨 노이즈) |

### 시나리오 테스트에서 확인된 추론 예시

홀드아웃 정확도는 데이터 규모 대비 높지 않았지만, 대표 시나리오에서는 높은 신뢰도로 올바른 방향을 가리켰다.

| 로그 (요약) | 예측 | 신뢰도 |
| --- | --- | --- |
| `java.lang.OutOfMemoryError: Java heap space` | Resource_Exhaustion | 94.4% |
| `IOException: No space left on device` | Disk_Full | 81.6% |
| `SocketTimeoutException: Read timed out` | Network_Timeout | 99.1% |
| `HikariPool-1 - Connection is not available` | DB_Pool_Exhaustion | 71.2% |
| `401 Unauthorized: Invalid API Key` | Auth_Failure | 66.2% |
| `Back-off restarting failed container` | Container_Crash | 55.0% |

## 한계 · 개선 방향

해커톤 PoC 단계에서 확인한 한계와 이후 확장 아이디어를 정리했다.

- **데이터 규모** — 181건으로는 10개 클래스 분류에 데이터가 부족해 홀드아웃 정확도가 50% 전후에 머물렀다. 실서비스 적용을 위해서는 로그·메트릭 페어 데이터의 지속적 축적이 필요하다.
- **메트릭 범위** — CPU·메모리·Latency 3종만 사용했다. 디스크 I/O, 네트워크 패킷 손실, GC 시간 등을 추가하면 자원 고갈·네트워크 장애 구분이 더 정교해질 수 있다.
- **시계열 문맥** — 단일 시점 스냅샷만 반영했다. 장애 직전 N분간의 메트릭 추이를 LSTM·Transformer로 넣으면 "갑자기 치솟은 CPU" 같은 패턴도 잡을 수 있다.
- **운영 연동** — Alertmanager, ELK, Prometheus 등 기존 관측 스택과 파이프라인을 연결하면 온콜 초기 트리아지 자동화로 확장 가능하다.

## 결과

AI-Enabled 해커톤 **구현 부문**에서 **AI를 활용한 자동 로그 분석 및 대응** 과제를 1인으로 제안·구현해 **대상**을 수상했다. 심사에서 높게 평가받은 포인트는 다음과 같다.

- 로그만 보는 기존 접근과 달리, **운영 현장의 실제 판단 방식**(로그 + 메트릭)을 모델 구조에 반영한 점
- 동일 로그·다른 메트릭 시나리오에서 **원인 추론이 달라지는 것**을 실험으로 명확히 보여준 점
- 자원 고갈, 네트워크, DB, 인증, 배포 실패 등 **운영 장애 유형을 체계적으로 분류**한 점

구현 코드는 `main.ipynb`에 정리되어 있으며, 데이터 로드 → 전처리 → 학습 → 시나리오 예측까지 한 번에 재현할 수 있다.
