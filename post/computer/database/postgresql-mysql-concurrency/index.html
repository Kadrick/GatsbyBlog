PostgreSQL vs MySQL — Online Judge를 위한 동시성 비교

채점 기록의 동시 읽기·쓰기를 위해 PostgreSQL을 검토한 과정 — MVCC, 격리 수준, 공식 문서 기반 정리

September 10, 2024

  • Computer
  • Database
  • PostgreSQL
  • MySQL

KNU Online Judge를 설계할 때 DBMS로 MySQL과 PostgreSQL을 함께 검토했다. 채점 기록 테이블에는 실습 마감 직전 수십~백 명이 동시에 제출하고, Worker가 결과를 쓰는 동안 학생·교수가 채점 결과를 조회하는 패턴이 반복된다. 이 글은 "동시 처리에 PostgreSQL이 유리하다"는 판단을 내리기 위해 공식 문서와 자료를 읽으며 정리한 내용이다. DB 내부를 완전히 이해한 결론이라기보다, 선택 근거를 검증하기 위한 학습 기록에 가깝다.

Online Judge에서 DB가 겪는 부하

Online Judge의 채점 기록 테이블에는 다음이 동시에 일어난다.

동작주체성격
제출 row INSERTAPI 서버쓰기
채점 결과 UPDATEJudge Worker쓰기
제출·채점 결과 SELECT학생, 교수/조교읽기

실습 마감 직전에는 쓰기가 집중되고, 그와 동시에 "내 제출 결과가 나왔는지" 확인하는 읽기도 이어진다. DB 입장에서는 읽기와 쓰기가 한 테이블에서 충돌하는 전형적인 다중 사용자(multiuser) 환경이다.

채점 실행 자체는 RabbitMQ + Worker로 API와 분리했지만(Message Queue 정리 글), 채점 결과를 DB에 기록하고 조회하는 단계는 여전히 DB의 동시성 처리에 의존한다.

MVCC란

MVCC(Multi-Version Concurrency Control)는 하나의 row에 여러 버전을 유지해, 읽기와 쓰기가 서로를 막지 않도록 하는 동시성 제어 방식이다. PostgreSQL과 MySQL InnoDB 모두 MVCC를 사용하지만, 구현 방식과 기본 격리 수준에서 차이가 난다.

PostgreSQL 공식 문서는 MVCC의 목적을 이렇게 설명한다.

PostgreSQL provides a rich set of tools for developers to manage concurrent access to data. Internally, data consistency is maintained by using a multiversion model (Multiversion Concurrency Control, MVCC).

The main advantage of using the MVCC model of concurrency control rather than locking is that in MVCC locks acquired for querying (reading) data do not conflict with locks acquired for writing data, and so reading never blocks writing and writing never blocks reading.

— PostgreSQL 18 Documentation — 13.1. Introduction

Severalnines 블로그는 두 DB의 MVCC 구현 차이를 다음과 같이 구분한다.

Most recognized approach to implements MVCC is the one used by PostgreSQL and Firebird/Interbase and another one is used by InnoDB and Oracle.

— Comparing Data Stores for PostgreSQL - MVCC vs InnoDB | Severalnines

PostgreSQL은 테이블 heap에 row 버전을 직접 보관하고, InnoDB는 undo log에 이전 버전을 저장하는 하이브리드 방식을 쓴다는 점이 핵심 차이 중 하나다.

PostgreSQL의 동시성 모델

기본 격리 수준: READ COMMITTED

PostgreSQL의 기본 격리 수준은 READ COMMITTED다.

Read Committed is the default isolation level in PostgreSQL. When a transaction uses this isolation level, a SELECT query (without a FOR UPDATE/SHARE clause) sees only data committed before the query began.

— PostgreSQL 18 Documentation — 13.2. Transaction Isolation

일반 SELECT는 스냅샷을 읽을 뿐 row에 잠금을 걸지 않는다. 쓰기 트랜잭션이 진행 중이어도, 읽기는 커밋된 이전 버전을 보며 서로 블로킹하지 않는다는 것이 PostgreSQL MVCC의 핵심이다.

Online Judge 관점

  • 학생이 제출 결과를 조회(SELECT)하는 동안 Worker가 다른 row를 UPDATE하거나 새 제출을 INSERT해도 읽기가 막히지 않는다.
  • 같은 row를 두 트랜잭션이 동시에 UPDATE하려는 경우는 여전히 충돌하지만, 읽기 vs 쓰기 간 경합은 MVCC로 완화된다.

MySQL InnoDB의 동시성 모델

기본 격리 수준: REPEATABLE READ

InnoDB의 기본 격리 수준은 REPEATABLE READ다.

The default isolation level for InnoDB is REPEATABLE READ.

— MySQL 8.4 Reference Manual — 17.7.2.1 Transaction Isolation Levels

REPEATABLE READ에서는 phantom read 방지를 위해 gap lock과 next-key lock이 사용된다.

For other search conditions, InnoDB locks the index range scanned, using gap locks or next-key locks to block insertions by other sessions into the gaps covered by the range.

— MySQL 8.4 Reference Manual — 17.7.2.1 Transaction Isolation Levels

MySQL 공식 문서는 next-key lock을 이렇게 정의한다.

A next-key lock is a combination of a record lock on the index record and a gap lock on the gap before the index record.

By default, InnoDB operates in REPEATABLE READ transaction isolation level. In this case, InnoDB uses next-key locks for searches and index scans, which prevents phantom rows.

— MySQL 8.4 Reference Manual — 17.7.1 InnoDB Locking

범위 조건으로 SELECT·UPDATE를 실행하면, 해당 인덱스 구간의 gap까지 잠글 수 있다. READ COMMITTED로 격리 수준을 낮추면 gap locking이 비활성화되지만, 그때는 phantom read가 발생할 수 있다.

Gap locking can be disabled explicitly. This occurs if you change the transaction isolation level to READ COMMITTED.

— MySQL 8.4 Reference Manual — 17.7.1 InnoDB Locking

Online Judge 관점

  • InnoDB도 MVCC를 쓰므로 "MySQL은 동시성이 안 된다"는 식의 단순 비교는 맞지 않다.
  • 다만 기본 REPEATABLE READ에서는 범위 스캔 시 gap/next-key lock이 걸릴 수 있고, MySQL 문서 자체도 REPEATABLE READ에서 deadlock 가능성을 언급한다.
  • 교수/조교가 기간·문제별로 넓은 범위를 조회하는 통계 쿼리와, 학생들의 동시 제출 INSERT가 겹치면 잠금 경합이 생길 여지가 있다.

Online Judge에서 PostgreSQL을 선택한 이유

위 내용을 읽고 내린 판단은 다음과 같다.

  1. 채점 기록은 읽기·쓰기가 동시에 몰리는 테이블이다. 실습 마감 직전 수십~백 명이 제출하고, 그와 동시에 결과 조회·통계 조회가 일어난다.
  2. PostgreSQL MVCC는 공식 문서 기준 "reading never blocks writing and writing never blocks reading" 을 보장한다.
  3. InnoDB도 MVCC를 쓰지만, 기본 REPEATABLE READ의 gap/next-key lock은 범위 조회가 많은 워크로드에서 잠금 경합·데드락 요인이 될 수 있다(MySQL Locking 문서).
  4. 다른 외주 프로젝트에서 MySQL을 써 본 경험은 있지만, Online Judge의 동시 제출 + 동시 조회 패턴에는 PostgreSQL이 더 맞다고 판단했다.

다만 이 판단에는 한계가 있다.

  • MySQL도 READ COMMITTED로 운영하면 gap lock을 줄일 수 있다. DB 선택만으로 성능이 결정되지는 않는다.
  • 인덱스 설계, 커넥션 풀, 쿼리 튜닝이 실제 병목에 더 큰 영향을 줄 수 있다.
  • 채점 실행 부하는 RabbitMQ로 분리했기 때문에, DB 동시성은 "제출·결과 기록·조회" 구간의 안정성에 집중해서 검토했다.

정리

Online Judge의 채점 기록은 단순 CRUD가 아니라 다수 사용자의 동시 읽기·쓰기가 겹치는 워크로드다. PostgreSQL과 MySQL InnoDB 모두 MVCC를 지원하지만, 기본 격리 수준과 잠금 전략이 다르다.

이 프로젝트에서는 PostgreSQL MVCC가 읽기·쓰기 비블로킹에 유리하다는 공식 문서 설명을 근거로 PostgreSQL을 선택했다. 완벽한 벤치마크 비교는 하지 않았고, 문서 기반의 설계 판단에 가깝다. 실제 적용 맥락은 KNU Online Judge 프로젝트 글에서 확인할 수 있다.
